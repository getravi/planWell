planWell

Architecture overview — last updated May 2026

1. System overview

planWell is a single-tenant financial planning tool deployed as a single Node.js process. The frontend is a React SPA served as static files by the same process. All state lives in one SQLite file, mounted on a Railway volume.

Browser └─ React SPA (Vite build, served by Hono) └─ fetch() → Hono API (:8787) ├─ Auth middleware (session cookie) ├─ /api/cube/* ← actuals + forecast reads ├─ /api/scenarios/* ← scenario CRUD + formula eval ├─ /api/analyst/* ← LLM Q&A + narrative ├─ /api/settings/* ← forecast horizon, AI model └─ /api/admin/* ← backup / restore │ └─ Repository (node:sqlite, synchronous) └─ planwell.sqlite ← Railway volume mount

2. Data model

The schema is a narrow-long planning cube with three dimension tables and two fact tables.

TableRoleKey columns
actuals Historical ledger month, department, account, value
forecast_values Computed forecast (denormalised) scenario_id, month, department, account, value
driver_assumptions User inputs per scenario / department / month scenario_id, scope_type, scope_key, month, driver_key, value
department / account Dimension hierarchies with sort order name, parent_name, sort_order
versions Named plan versions (actuals or scenario) id, name, kind, locked
custom_variables User-defined KPIs (input or calculated) id, label, kind, formula, sort_order
app_settings Key-value store for settings key, value

Design decisions

  • Denormalised forecast cache. forecast_values stores computed outputs rather than re-running the formula engine on every read. This makes cube queries fast and keeps the read path simple. The trade-off is that assumptions changes must trigger a recalculateAllScenarios() write pass.
  • Narrow-long layout. Every (month, department, account) triplet is a row. This makes arbitrary slice-and-dice queries trivial SQL aggregations and avoids the wide-table sparse-column problem common in spreadsheet-style schemas.
  • Driver assumptions separate from outputs. Keeping inputs (driver_assumptions) and outputs (forecast_values) in separate tables means the formula engine is always re-derivable from inputs, and rollback/audit is straightforward.
  • Composite primary keys on fact tables. Indexed on the most common access pattern: (scenario_id, month, department, account).

3. Technology choices

LayerChoiceRationale
Runtime Node.js 24 Built-in node:sqlite (no native addon to compile), --env-file-if-present, and fast startup on Railway.
Server framework Hono Minimal, typed, zero-dependency routing. Serves both the API and the built SPA from one process with no Nginx layer.
Database SQLite via node:sqlite Single-file, zero-ops, no connection pool. Synchronous API eliminates async/await overhead for hot read paths. Backup is a single database.serialize() call.
Frontend React 19 + Vite React 19 concurrent features for responsive UI; Vite for fast HMR and lean prod bundles.
Data fetching TanStack Query Cache invalidation, background refetch, and loading states without a global store. Query keys map 1-to-1 with API routes.
Formula engine mathjs Safe sandboxed expression evaluation for user-defined COGS/OpEx formulas and custom variables.
Validation Zod Runtime schema validation at every API boundary; shared types inferred by TypeScript on the server side.
Deployment Railway Volume mounts for SQLite persistence across deploys, PORT env var injection, and simple env-var secrets management.

4. AI / ML approach

Provider strategy

Two LLM providers are supported. The active provider is determined by which API key is set:

  • Anthropic Claude — agentic tool-use loop (preferred)
  • Google Gemini — single-pass context injection (fallback)

The selected model is stored in app_settings and loaded at call time, so it can be changed from the Admin UI without a restart.

Grounded Q&A (Claude path)

Claude is given four tools over the planning cube and forced to call at least one before answering (tool_choice: { type: "any" } on the first turn). The agentic loop runs up to 5 iterations, accumulating tool results before producing a final answer.

User question └─ ClaudeAnalyst.ask() ├─ system: "Available scenarios: Base, Upside, …" ├─ tool_choice: any (first turn — forces grounding) │ ├─ iteration 1 → Claude calls getMetricSummary("Base") ├─ iteration 2 → Claude calls compareScenarios("Base","Upside") └─ iteration 3 → stop_reason: end_turn → return answer

Multi-turn conversation history is passed in full on every request so follow-up questions have context.

Grounded Q&A (Gemini path)

Because the Google GenAI SDK does not support the same agentic tool-use loop, Gemini receives a single prompt containing all scenario summaries pre-fetched from the database. This trades token efficiency for reliability.

Narrative generation

A separate POST /api/analyst/narrative endpoint builds a rich context object (KPI summary, top variance rows, top anomaly flags) and asks the LLM to return structured JSON matching { headline, sections[], risks[] }. The system prompt instructs the model not to use markdown fences; a fence-stripping fallback handles non-compliant responses.

Anomaly detection

Pure statistical, no model inference. For each (department, account) time series: Z-score flagging (|value − mean| > 2σ) and month-over-month spike detection (> 25% change). Results surface as amber indicators in the Actuals grid and are fed to the narrative prompt as context.

Baseline suggestions

Driver defaults are computed from actuals using CAGR for Revenue and Headcount, trailing-average COGS % of Revenue, and trailing cost-per-head (OpEx ÷ Headcount). All deterministic — no LLM involved.

Grounding rationale: LLMs are forbidden from answering from training knowledge. The Q&A path forces at least one tool call so every number in the answer traces back to data in the database. This makes the analyst auditable — wrong answers point to wrong data, not model hallucination.

AreaCurrent stateBetter approach
Multi-tenancy Single SQLite file, single user account Schema-per-tenant or separate SQLite files per org; session scoped to tenant ID
Formula engine security mathjs sandboxed eval — good but not isolated Run formula evaluation in a Worker thread or Wasm sandbox to prevent prototype pollution
Forecast recalculation Synchronous full-pass on every assumption change Background job queue; only recalculate affected (scenario, department) combos; push progress via SSE
Gemini analyst Dumps all scenario summaries into one prompt; no tool-use loop Implement proper agentic loop via Gemini function calling once the SDK stabilises; share the same tool definitions as Claude
LLM citations Fixed citations (revenue + gross margin) regardless of question Parse tool calls from the Claude message history to build citations dynamically from actual tool results
Testing Minimal — one domain test file Integration tests against a real SQLite fixture; MSW for API mocking in frontend tests; property-based tests for the formula engine
