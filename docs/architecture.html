planWell — Architecture

Financial planning tool. Single-file SQLite, Hono API, React 19 SPA, optional LLM analyst.

1. Data model

The schema is a star schema: dimension tables (time_month, department, account) and two fact tables (actuals, forecast_values). Planning intent lives in scenarios / scenario_formulas / custom_variable_values.

TableRoleKey design decision
actuals Raw imported financials — month × dept × account × value. No UNIQUE constraint on the tuple. replaceActuals() deletes all rows and re-inserts; the constraint is enforced at the application layer, not the DB layer. Keeps the import path simple.
forecast_values Pre-computed output of the forecast engine — scenario × month × dept × account × value. Same no-UNIQUE decision for the same reason. Recalculated fully per scenario on every assumption change (background setImmediate after save).
versions Tracks planning versions. Each scenario IS a version (kind = 'scenario'). Actuals use a reserved version (kind = 'actuals'). Unifying actuals and scenarios under versions lets locking, ordering, and copy-on-fork logic share one code path.
custom_variables + custom_variable_values User-defined driver inputs and calculated expressions. Values stored with a scope string encoding (dept, month, or both). Scope as a string key avoids a four-column composite PK. Resolution follows a 6-level precedence chain (default → global → monthly → ancestor dept global → ancestor dept monthly → dept monthly).

2. Technology choices

TechnologyWhy
Node.js 24 node:sqlite (DatabaseSync) Zero-dependency, synchronous SQLite built into Node 24. No separate process, no async complexity, no driver to maintain. Single writer is fine for a planning tool with one active user session. The synchronous API makes background recalculation via setImmediate straightforward — Node.js is single-threaded, so no write conflicts are possible.
Hono 4 Lightweight (~13 KB), Web-standard Request/Response, runs on Cloudflare Workers and Node equally. Built-in middleware for cookies, CORS, and security headers. No magic — routing and middleware are explicit and easy to test.
React 19 + TanStack Query v5 React for component model. TanStack Query for server-state caching and invalidation — particularly useful here because the SSE recalc-done event triggers targeted query invalidations without prop drilling.
math.js Safe expression evaluator with a sandboxable scope. Used for user-authored forecast formulas. Provides topological-sort-friendly AST traversal for custom variable dependency analysis, and guards against unsafe globals via the import block.
Anthropic SDK + Google GenAI SDK Both support agentic (tool-use) loops. Provider is selected at startup: Anthropic if ANTHROPIC_API_KEY is set, Gemini if GEMINI_API_KEY is set, otherwise local rule-based fallback. No UI required to switch — set the env var.
Pino Structured JSON logging. All HTTP requests, background recalc events, and formula errors emit structured log lines — useful on Railway where logs are piped to a log aggregator.
Vite + vite-plus Fast HMR in dev, optimized bundle in prod. vite-plus bundles Vitest and oxlint — one config file covers build, test, and lint.

3. AI / ML approach

Three layers, in ascending capability order:

Local fallback (always available)

Rule-based pattern matching against the user's question. Answers questions about revenue, COGS, headcount, and OpEx using getMetricSummary(). No latency, no API cost, works offline. Answers are short and deterministic.

Gemini (Google GenAI SDK)

Agentic loop with four tools: getMetricSummary, listActuals, compareScenarios, detectAnomalies. The model decides which tools to call and synthesizes a grounded answer with dynamic citations. Activated by GEMINI_API_KEY.

Claude (Anthropic SDK)

Same four tools, same agentic loop pattern. Activated by ANTHROPIC_API_KEY (takes priority over Gemini). Uses prompt caching on the system prompt and tool definitions to reduce latency on repeated questions. Uses claude-sonnet-4-6 by default; model is overridable via the Settings page.

Baseline suggestions (no LLM)

GET /api/forecast/baseline-suggestions computes suggested driver values (revenue growth rate, COGS %, headcount growth, cost per head) from trailing actuals using CAGR and rolling averages — no LLM required. The "Suggest from actuals" button populates the driver matrix from these values. Users can accept, reject, or edit before saving.

4. What I'd do differently with more time

AreaCurrent stateBetter approach
Multi-tenancy Single SQLite file, one user account per deployment Schema-per-tenant or separate SQLite files per org; scoped sessions; invite flow for collaborators
Partial recalculation Every save recalculates the full scenario across all departments and months Track which (dept, month) cells are dirty and only recompute those. Significant win at 50+ departments × 24 months.
Horizontal scaling recalcSubs is in-process — SSE only works with a single server instance Replace the in-memory pub/sub with Redis pub/sub or a database-polled queue. Each instance subscribes and fans out to its own connections.
Testing Integration tests only (Hono + SQLite in-memory). No property-based tests, no MSW for client tests. Add fast-check for formula engine edge cases. Use MSW in client tests instead of mocked fetch — tests would survive API shape changes.
LLM cost controls No rate limiting, token counting, or cost cap on analyst queries Track monthly spend via the API provider's usage endpoint; surface a warning in the UI when approaching a configured limit.
Auth Basic email/password, no rate limiting on login, no 2FA OAuth (Google/GitHub) via a provider like Auth.js; add login rate limiting and account lockout immediately.
